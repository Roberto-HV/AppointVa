5. Registro de Cambios — Sesión Actual (Junio 2026)

v2.2 — Observabilidad, rendimiento y política de cancelación

🏥
Health check endpoint en /health
Endpoint GET /health que verifica la conexión a la base de datos (EF Core DbContext check). Responde con JSON estructurado:
{ "estado": "Healthy", "checks": [{ "nombre": "base-de-datos", "estado": "Healthy" }] }
Necesario para Railway, Render, Azure App Service y cualquier orquestador que requiera readiness probe.
Program.cs Microsoft.Extensions.Diagnostics.HealthChecks.EntityFrameworkCore 8.0.10
📝
Serilog — logging estructurado persistente
Reemplaza el logger genérico de .NET por Serilog con doble sink: consola (formato legible) y archivo con rolling diario en logs/appointva-YYYYMMDD.log. Configurado con nivel mínimo Warning para no saturar el disco con ruido del framework, capturando todos los errores de la aplicación.
• Rotación automática diaria, retención de últimos 30 archivos.
• Se resolvió conflicto de nombre Resend.Log vs Serilog.Log usando el namespace completo Serilog.Log.Logger = ...
Program.cs Serilog.AspNetCore 8.0.3
⚡
Output cache en GET /publico/negocios/{slug}
El endpoint más golpeado del sistema (se llama cada vez que un cliente abre la página de booking) ahora se cachea 5 minutos con [OutputCache(Duration = 300)] nativo de .NET 8. Incluye empleados, servicios, categorías y colores — datos que cambian muy poco.
Impacto: 50 visitas/hora → 1 query a DB cada 5 min en lugar de 50.
app.UseOutputCache() agregado al pipeline HTTP.
Program.cs PublicoController.cs
⏱️
Política de cancelación configurable
Los negocios ahora pueden definir cuántas horas de anticipación mínima requieren para cancelar una cita.
Backend:
• Campo HorasCancelacion int (default 0 = sin restricción) añadido al modelo Negocio.
• Migración EF Core PoliticaCancelacion generada.
• PUT /api/negocios/perfil acepta y persiste el nuevo campo (ActualizarNegocioDto).
• DELETE /publico/citas/{codigo} valida que cita.InicioEn - UtcNow > HorasCancelacion antes de cancelar. Retorna 400 con mensaje legible si se viola la política.
• GET /publico/citas/{codigo} devuelve horasCancelacion en el DTO.
Frontend:
• PerfilPage: nuevo select "Política de cancelación" con opciones 0h–48h.
• ConfirmacionPage: muestra aviso de política, oculta el botón "Cancelar cita" si ya no es posible cancelar según el tiempo restante. Calcula en cliente para feedback inmediato.

Transacciones Serializables en reservas (anti race condition)
Tanto CitasController.Crear como PublicoController.CrearCita envuelven el check de solapamiento + inserción en una transacción IsolationLevel.Serializable. Si dos reservas concurrentes chocan, PostgreSQL lanza el error 40001 y se retorna 409 Conflict. Helper estático EsConflictoSerializacion detecta el tipo de excepción de Npgsql sin depender del tipo concreto.
Además se valida que el empleado pertenezca al negocio antes de crear la cita, evitando booking cross-business.

2FA — Backend

🔐
2FA con TOTP — 5 endpoints + login modificado
Usa el soporte nativo de TOTP de ASP.NET Identity (sin paquetes externos).
Login: si el usuario tiene 2FA activo, retorna {requiere2FA: true, challengeToken}. El challenge token es un JWT especial con claim "2fa_challenge": "true", expira en 5 minutos.
Nuevos endpoints:
• GET /auth/2fa/estado — {habilitado, tieneConfiguracion}
• POST /auth/2fa/configurar — genera/resetea la clave TOTP y retorna {uri: "otpauth://...", llave: "JBSW Y3DP..."}
• POST /auth/2fa/activar — verifica código TOTP y llama SetTwoFactorEnabledAsync(true)
• POST /auth/2fa/desactivar — verifica código y resetea la clave TOTP
• POST /auth/2fa/verificar — valida challenge token + código TOTP, retorna JWT completo
IJwtService: nuevos métodos GenerarChallengeToken2FA y ValidarChallengeToken2FA.

9. Robustez del Sistema y Mejoras Sugeridas

¿Qué tan robusto es para 10–20 negocios clientes?

Veredicto: Muy sólido para esta escala

Un servidor .NET 8 puede manejar miles de solicitudes/segundo. Con 10-20 negocios, carga estimada: 20-100 usuarios simultáneos en hora pico, 500-4,000 reservas/mes totales, 40-80 requests/minuto de polling del dashboard. Márgenes muy amplios.

En v2.2 se implementaron 4 de las 7 mejoras prioritarias: Output cache, Serilog, Health check y Política de cancelación. El sistema está preparado para un deploy a producción real.

Mejoras recomendadas para escalar a 10-20 clientes en producción

⚡
1. Output cache en la página pública de reservas ✅ Implementado v2.2
GET /api/publico/negocios/{slug} se ejecuta cada vez que un cliente abre la página de booking. Incluye empleados, servicios y colores — datos que cambian poco. Un cache de 5 minutos con [OutputCache(Duration = 300)] (nativo en .NET 8) elimina prácticamente toda la carga de consulta en horas pico.
Impacto: Si un negocio tiene 50 visitas/hora, reduce 50 queries/hora a 1.
📋
2. Logging estructurado con Serilog ✅ Implementado v2.2
Configurado con rolling file diario en logs/, retención 30 días, nivel Warning mínimo. Consola con template legible. Crítico para diagnóstico en producción cuando emails fallan o hay errores inesperados en reservas.
🔁
3. Retry en jobs de Hangfire si el email falla
Hangfire ya tiene reintentos automáticos (10 por defecto con backoff exponencial). Pero si el proveedor Resend está caído en el momento del recordatorio, el job falla y ya no re-intenta cuando Resend vuelve.
Solución: configurar AutomaticRetryAttribute con más reintentos y alertas para jobs que fallen definitivamente. También considerar DisableConcurrentExecution en el job de recordatorios para evitar duplicados.
💳
4. Validación del límite de citas por plan
El modelo de negocio define planes con límites (Básico: 200 citas/mes, Pro: 1,000, Premium: 10,000) pero ningún endpoint verifica si el negocio ha superado su límite. Un negocio en plan Básico puede crear citas ilimitadamente.
Solución: al crear cita en CitasController y PublicoController, consultar COUNT(*) de citas del mes y comparar con negocio.Plan.LimiteCitasMes. Retornar 402 Payment Required si se supera.
⏱️
5. Política de cancelación configurable ✅ Implementado v2.2
Campo HorasCancelacion en Negocio (default 0 = sin restricción). CancelarCita valida que InicioEn - UtcNow > HorasCancelacion. Configurable desde PerfilPage. Frontend muestra aviso y oculta el botón de cancelar si ya venció la ventana.
🏥
6. Health check endpoint ✅ Implementado v2.2
GET /health verifica la conexión a la DB y devuelve JSON estructurado. Necesario para Railway, Render, Azure App Service y cualquier orquestador que requiera readiness probe.
📱
7. Enlace de WhatsApp al confirmar cita ✅ Implementado
Botón "Compartir por WhatsApp" en ConfirmacionPage con mensaje prearmado (negocio, servicio, empleado, fecha, link). Sin API externa — solo enlace https://wa.me/?text=....
